# ToyC Compiler

一个面向课程实践的 ToyC 编译器项目，目标是将 ToyC 源程序编译为可正确执行的 **RISC-V32 汇编代码**。本仓库的设计依据来自课程任务要求与四人分工方案，当前 README 用于统一项目目标、模块边界、开发流程与交付标准。

## 项目目标

本项目需要完成一条完整的编译流水线：

1. 读取 `stdin` 中的 ToyC 源代码
2. 进行词法分析、语法分析、语义分析
3. 生成与目标机器无关的中间代码 IR
4. 将 IR 翻译为 RISC-V32 汇编
5. 将汇编输出到 `stdout`

除编译器本体外，项目还需要提交一份实践报告，并通过在线评测系统的功能测试与性能测试。

## 任务背景

ToyC 是 C 语言的一个简化子集，去掉了数组、指针、I/O、多文件编译等特性，适合作为编译器课程项目。根据本年度任务要求，ToyC **新增了全局变量与全局常量支持**，这也是本项目实现中的重点。

评测时：

- 编译器从标准输入读取 ToyC 程序
- 编译器向标准输出输出 RISC-V32 汇编
- 可接受可选参数 `-opt` 以启用优化
- 评测结果同时考虑功能正确性、性能和实践报告

## 评分方式

课程文档给出的评分方式如下：

- 总得分 = 评测得分 × 80% + 实践报告得分 × 20%
- 评测得分 = 功能得分 × 75% + 性能得分 × 25%

单个测试点限制：

- 功能测试：1 秒
- 性能测试：20 秒

因此本项目的实现策略应当是：

1. 先保证前后端联调正确，尽可能提高功能测试通过率
2. 再在 `-opt` 模式下逐步加入中端和后端优化

## 技术路线

根据任务要求，项目可以使用以下技术栈之一：

- C++20 + CMake / Make
- Java 21 + Maven / Gradle

结合分工方案，本项目更适合采用 **C++20 + CMake**，理由如下：

- 更适合编译器课程常见的数据结构与模块组织
- 便于实现 AST、符号表、IR、寄存器分配等底层结构
- 更容易以单一命令行程序形式适配评测环境

## ToyC 语言范围

### 支持的程序结构

- 全局变量声明
- 全局常量声明
- 函数定义
- 入口函数 `int main()`

### 支持的语句

- 语句块
- 空语句
- 表达式语句
- 赋值语句
- 局部变量/常量声明
- `if-else`
- `while`
- `break`
- `continue`
- `return`

### 支持的表达式

- 整数字面量
- 标识符引用
- 括号表达式
- 函数调用
- 一元运算：`+` `-` `!`
- 二元运算：`*` `/` `%` `+` `-`
- 关系运算：`<` `>` `<=` `>=` `==` `!=`
- 逻辑运算：`&&` `||`

### 关键语义约束

- 程序必须包含 `int main()`
- 所有变量和常量声明都必须带初始化表达式
- `const` 初值必须能在编译期确定
- 常量不能再被赋值
- 标识符必须先声明后使用
- 支持块级作用域与同名遮蔽
- `break` / `continue` 只能出现在循环中
- `int` 函数必须保证所有可能路径都有返回值
- `void` 函数若使用 `return`，则不能带返回值
- 逻辑与、逻辑或必须遵守短路求值

## 编译器总体架构

项目采用四层流水线结构：

```text
ToyC Source
   ↓
Frontend
   Token + AST
   ↓
Semantic Analysis
   已绑定符号并完成语义标注的 AST
   ↓
IR / Optimizer
   机器无关中间代码
   ↓
RISC-V32 Backend
   汇编输出
```

对应模块职责如下：

- 前端：词法分析、语法分析、AST 构建
- 语义层：符号表、作用域、类型信息、常量求值、控制语义
- 中端：IR 生成、CFG 构建、短路求值、优化
- 后端：RISC-V32 指令选择、栈帧布局、寄存器分配、指令级优化

## 模块接口约定

为了支持并行开发，四层之间只通过稳定的数据结构交互。

### A → B

A 输出 AST 根节点：

```cpp
std::unique_ptr<CompUnitAST> parseProgram();
```

B 只能读取并标注 AST，不依赖 Parser 内部状态。

### B → C

B 为标识符和声明完成符号绑定：

```cpp
struct IdentifierExpr {
    std::string name;
    Symbol* resolvedSymbol;
};
```

C 不再通过字符串名称查找变量，而是直接使用 `resolvedSymbol` 和语义属性。

### C → D

C 输出完整 IR 模块：

```cpp
struct IRModule {
    std::vector<IRGlobal> globals;
    std::vector<IRFunction> functions;
};
```

D 只依赖 IR 公共接口，不直接操作 AST 和符号表。

### 优化开关

命令行约定：

```text
compiler
compiler -opt
```

- 未传 `-opt`：输出正确优先版本
- 传入 `-opt`：启用中端和后端优化

## 测试策略

### 分层测试

建议按模块分层测试：

- Lexer 测试
- Parser / AST 测试
- 语义分析测试
- IR 测试
- 后端测试
- 端到端测试
- 性能测试

### 差分测试

由于 ToyC 是 C 的子集，可以对同一份 ToyC 程序：

1. 用本项目编译为 RISC-V32 汇编并运行
2. 用 `gcc` 编译等价 C 程序并运行
3. 比较最终返回值

这会是最有效的正确性验证方式之一。

### 重点测试场景

- 多层嵌套作用域
- 全局变量与局部变量同名
- `const` 引用其他常量
- 递归函数
- `void` / `int` 函数调用差异
- `break` / `continue` / `return` 组合
- 短路求值是否跳过函数调用
- 优化前后返回值一致性

## 设计重点与难点

本项目最重要、也最容易出错的点包括：

- `const` 的编译期求值
- 全局变量和全局常量支持
- 短路逻辑的正确实现
- `int` 函数所有路径返回分析
- 函数调用约定与栈帧布局
- 优化前后程序行为一致

建议优先保证以下四项正确：

1. 作用域与符号绑定
2. 全局对象管理
3. 短路求值
4. 函数调用与返回值处理

在这几项稳定后，再逐步增强优化能力。

## 优化目标

根据课程评分方式，优化不是可有可无的附加项，而是影响评测成绩的重要部分。建议在 `-opt` 模式下逐步加入：

- 常量折叠
- 常量传播
- 代数化简
- 死代码删除
- 无效跳转删除
- 条件分支简化
- 局部公共子表达式消除
- 基本块合并
- 简单复制传播
- 指令级 peephole 优化
- 寄存器分配优化

优化策略应遵循“先正确、后加速”的原则，避免为了性能损失功能测试通过率。
