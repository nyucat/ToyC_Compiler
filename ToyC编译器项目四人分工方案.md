# ToyC 编译器项目四人分工方案

## 一、总体分工原则

项目采用“前端—语义—中端—后端”四层架构，将四位成员分别安排在相对独立的模块中：

| 成员 | 核心模块 | 辅助职责 | 预计工作量 |
|---|---|---|---:|
| A | 词法分析、语法分析、AST | 工程框架、自动化测试平台 | 约25% |
| B | 语义分析、符号表、常量求值 | 全局对象管理、语义测试 | 约25% |
| C | 中间代码生成、控制流、优化 | IR调试工具、优化测试 | 约25% |
| D | RISC-V32后端代码生成 | 寄存器分配、性能测试 | 约25% |

四个模块之间依次通过统一的数据结构连接：

```text
ToyC源代码
   ↓
A：Token序列、AST
   ↓
B：完成符号绑定和语义标注的AST
   ↓
C：自定义中间代码IR
   ↓
D：RISC-V32汇编代码
```

项目初期需要共同确定 AST 和中间代码 IR 的接口。接口确定后原则上不随意修改，各成员只在自己的目录中开发，从而减少多人同时修改同一文件造成的冲突。

## 二、成员A：词法分析、语法分析与工程基础设施

### 1. 核心负责内容

A负责整个编译器前端，包括从标准输入读取ToyC源程序，将源程序转换为抽象语法树AST。

具体工作包括：

1. 设计并实现词法分析器，识别ToyC的全部词法单元。
2. 识别关键字，包括`const`、`int`、`void`、`if`、`else`、`while`、`break`、`continue`和`return`。
3. 识别标识符、整数、运算符、分隔符和括号。
4. 正确跳过空格、制表符、换行符、单行注释和多行注释。
5. 处理`+`、`-`、`*`、`/`、`%`、`!`、`&&`、`||`、`==`、`!=`、`<=`、`>=`等运算符。
6. 设计Token类型、Token值以及源代码位置信息。
7. 使用递归下降、Flex/Bison或ANTLR实现语法分析器。
8. 按照运算符优先级和结合性解析表达式。
9. 解析全局声明、函数定义、形参、语句块和各类控制语句。
10. 构建统一的AST节点体系，供语义分析和中间代码生成使用。

### 2. AST设计职责

A需要设计以下主要AST节点：

```text
CompUnit
├── VarDecl
├── ConstDecl
└── FuncDef
    ├── Param
    └── Block
        ├── DeclStmt
        ├── AssignStmt
        ├── ExprStmt
        ├── IfStmt
        ├── WhileStmt
        ├── BreakStmt
        ├── ContinueStmt
        └── ReturnStmt
```

表达式节点至少包括：

```text
NumberExpr
IdentifierExpr
UnaryExpr
BinaryExpr
CallExpr
```

AST节点只保存语法结构，不应直接包含RISC-V寄存器、栈偏移等后端信息，避免前端和后端相互依赖。

### 3. 工程与测试辅助职责

为了平衡工作量，A同时负责项目的基础工程环境：

1. 建立项目目录结构。
2. 编写`CMakeLists.txt`或其他构建文件。
3. 实现编译器主程序入口。
4. 支持从`stdin`读取ToyC程序、向`stdout`输出汇编。
5. 识别命令行参数`-opt`，并将优化开关传递给中端。
6. 编写统一测试脚本。
7. 建立差分测试流程：同一份ToyC程序分别由gcc和本组编译器编译，比较最终返回值。
8. 配置Git仓库、忽略文件以及基础CI检查。
9. 负责最终项目构建方式和在线评测环境兼容性检查。

### 4. 主要交付物

```text
src/frontend/lexer.*
src/frontend/parser.*
src/ast/ast.*
src/main.*
CMakeLists.txt
tests/scripts/
```

A向B提供完整AST。只要AST公共接口保持稳定，B、C、D不需要了解词法和语法分析的内部实现。

## 三、成员B：语义分析、符号表与常量求值

### 1. 核心负责内容

B负责对A生成的AST进行语义处理，并为AST中的标识符、函数调用和声明补充必要的语义信息。

主要工作包括：

1. 设计符号表和作用域管理机制。
2. 管理全局作用域、函数作用域和嵌套语句块作用域。
3. 支持内层作用域屏蔽外层同名变量、常量和形参。
4. 区分局部变量、全局变量、局部常量、全局常量、函数和函数形参。
5. 为每个声明生成唯一的符号对象或符号编号。
6. 将每个标识符引用绑定到其对应的声明。
7. 记录函数返回类型、形参数量和形参类型。
8. 处理函数调用关系，并确认被调用函数已经声明。
9. 检查`main`函数是否满足无参数且返回类型为`int`。
10. 识别变量是左值还是右值。

虽然正式测试程序满足语义约束，但符号绑定、作用域解析和类型信息仍是后续代码生成所必需的，因此不能完全省略语义分析模块。

### 2. 常量与声明处理

B重点负责本年度新增的全局变量和常量功能：

1. 实现编译期常量表达式求值。
2. 支持数字、已声明常量和各种算术、关系、逻辑运算组成的常量表达式。
3. 计算并保存常量的最终整数值。
4. 标记常量不可被赋值。
5. 记录全局变量的初始化值。
6. 为全局变量、全局常量和局部对象生成不同的存储类别。
7. 为后端提供全局对象列表及其初始值。
8. 处理同名变量在不同作用域下的唯一标识问题。

### 3. 控制语义处理

B还负责以下控制语义信息：

1. 判断`break`和`continue`是否位于循环内部。
2. 处理`int`函数和`void`函数的返回值差异。
3. 标记返回表达式的语义类型。
4. 检查无返回值函数调用是否被错误地当作表达式值使用。
5. 对函数返回路径进行基本分析。
6. 为循环和函数建立语义上下文。
7. 为C的中间代码生成提供已经解析完成的符号引用。

### 4. 主要数据结构

建议设计统一符号结构：

```cpp
struct Symbol {
    int id;
    std::string name;
    SymbolKind kind;       // Variable、Constant、Function、Parameter
    ValueType type;        // Int、Void
    StorageClass storage;  // Global、Local、Parameter
    bool isConst;
    int constValue;
};
```

符号表可采用作用域栈：

```text
全局符号表
   ↓ 进入函数
函数作用域
   ↓ 进入语句块
局部作用域1
   ↓ 进入内层语句块
局部作用域2
```

### 5. 测试职责

B负责设计以下测试：

1. 全局变量和局部变量同名。
2. 多层语句块变量屏蔽。
3. 常量引用其他常量。
4. 函数参数和局部变量同名。
5. 递归函数调用。
6. `int`与`void`函数调用。
7. 多个全局变量和函数混合声明。
8. `break`、`continue`和`return`的组合场景。

### 6. 主要交付物

```text
src/sema/symbol.*
src/sema/scope.*
src/sema/semantic_analyzer.*
src/sema/constant_evaluator.*
tests/sema/
```

B向C提供“已经完成符号绑定和语义标注的AST”，C只使用符号编号和语义属性，不再通过变量名称查找声明。

## 四、成员C：中间代码生成、控制流构建与中端优化

### 1. 核心负责内容

C负责将语义分析后的AST转换成结构清晰、与RISC-V无关的中间代码IR。

推荐采用三地址码或虚拟寄存器形式，例如：

```text
%1 = const 10
%2 = load @a
%3 = add %1, %2
store %3, @b
br %3, label_true, label_false
call foo(%1, %2)
ret %3
```

C需要设计的中间代码包括：

1. 常量加载。
2. 局部变量和全局变量的读取、写入。
3. 一元运算和二元运算。
4. 比较运算。
5. 无条件跳转和条件跳转。
6. 函数调用和参数传递。
7. 函数返回。
8. 基本块、函数和全局对象表示。
9. 虚拟寄存器编号。
10. 标签和控制流边。

### 2. AST到IR转换

C负责实现各种语法结构的中间代码生成：

1. 普通算术表达式。
2. 关系表达式。
3. 逻辑表达式。
4. 变量赋值。
5. 变量和常量初始化。
6. `if`语句和`if-else`语句。
7. `while`循环。
8. `break`和`continue`。
9. 函数定义。
10. 函数调用和递归调用。
11. `return`语句。
12. 多层嵌套控制结构。

### 3. 短路求值

逻辑与和逻辑或必须符合C语言短路求值规则，C负责通过基本块和条件跳转实现，例如：

```c
a && func()
```

当`a`为0时不能调用`func()`。

```c
a || func()
```

当`a`非0时不能调用`func()`。

这部分应在IR阶段完成，而不是交给后端临时判断。

### 4. 中端优化

项目性能得分占评测成绩的25%，因此C需要围绕`-opt`实现若干中端优化：

1. 常量折叠。
2. 常量传播。
3. 代数化简。
4. 无法到达基本块删除。
5. 死代码删除。
6. 无效跳转删除。
7. 条件分支简化。
8. 局部公共子表达式消除。
9. 基本块合并。
10. 简单复制传播。

例如：

```text
%1 = const 2
%2 = const 3
%3 = mul %1, %2
```

可以优化为：

```text
%3 = const 6
```

再如：

```text
br 0, block1, block2
```

可以直接优化为：

```text
jump block2
```

应首先保证未开启优化时能够正确运行，再逐步加入优化，避免优化错误影响功能测试通过率。

### 5. IR调试与测试职责

C需要实现IR文本输出工具，使每条ToyC程序都可以输出可读的中间代码，方便定位问题。

测试重点包括：

1. 多层`if-else`。
2. 多层`while`。
3. 循环中的`break`和`continue`。
4. 短路求值是否跳过函数调用。
5. 递归函数的调用关系。
6. 常量传播和死代码删除。
7. 优化前后程序返回值一致。
8. 不同优化Pass连续执行时是否保持正确性。

### 6. 主要交付物

```text
src/ir/ir.*
src/ir/ir_builder.*
src/ir/basic_block.*
src/optimizer/constant_fold.*
src/optimizer/dead_code.*
src/optimizer/cfg_simplify.*
tests/ir/
tests/optimization/
```

C向D提供稳定的IR。D只负责翻译IR，不需要重新处理AST、作用域或短路逻辑。

## 五、成员D：RISC-V32后端、寄存器分配与性能优化

### 1. 核心负责内容

D负责将C生成的IR转换为可以被评测环境编译和执行的RISC-V32汇编代码。

主要工作包括：

1. 设计RISC-V32汇编指令表示。
2. 将IR算术运算映射为RISC-V指令。
3. 实现整数加减乘除和取模。
4. 实现关系比较和逻辑运算。
5. 实现条件分支和无条件跳转。
6. 输出函数标签、局部标签和全局符号。
7. 处理全局变量对应的`.data`或其他数据段。
8. 处理只读常量或立即数。
9. 生成符合工具链要求的完整汇编文本。
10. 保证`main`函数返回值能够作为程序退出代码返回。

### 2. 函数调用约定

D重点负责函数调用和栈帧管理：

1. 按照RISC-V调用约定传递函数参数。
2. 使用`a0`至`a7`处理前若干个参数。
3. 支持参数数量超过参数寄存器数量的情况。
4. 使用`a0`保存函数返回值。
5. 保存和恢复返回地址`ra`。
6. 正确调整`sp`。
7. 为局部变量、临时值和保存寄存器分配栈空间。
8. 处理调用者保存寄存器和被调用者保存寄存器。
9. 处理普通函数调用和递归函数调用。
10. 保证栈帧大小和栈对齐满足要求。

建议生成类似结构：

```asm
func:
    addi sp, sp, -frame_size
    sw ra, offset(sp)
    sw s0, offset(sp)

    # 函数主体

    lw s0, offset(sp)
    lw ra, offset(sp)
    addi sp, sp, frame_size
    ret
```

### 3. 寄存器分配

为降低开发风险，可以分两步实现：

第一阶段采用正确性优先方案，将虚拟寄存器保存到栈中，使用少量临时寄存器完成运算，优先保证所有功能测试通过。

第二阶段实现性能优化，例如：

1. 基本块内寄存器复用。
2. 活跃变量分析。
3. 线性扫描寄存器分配。
4. 减少重复的`lw`和`sw`。
5. 优先将高频变量放入物理寄存器。
6. 函数调用前后仅保存必要寄存器。
7. 减少不必要的栈空间。

### 4. 后端优化

D负责面向RISC-V指令的局部优化：

1. 删除无效的`mv x, x`。
2. 删除跳转到下一条指令的`j`。
3. 合并连续的栈指针调整。
4. 消除重复加载。
5. 使用立即数指令代替不必要的常量加载。
6. 减少连续的内存读写。
7. 简化函数序言和函数尾声。
8. 对叶子函数避免保存不需要的寄存器。

### 5. 性能测试职责

D负责建立性能测试记录：

1. 统计生成汇编的指令数量。
2. 比较开启和关闭`-opt`时的运行时间。
3. 对循环、递归和大量函数调用程序进行测试。
4. 找出频繁访存或频繁保存寄存器的问题。
5. 将本组程序运行时间与gcc生成结果进行对比。
6. 汇总优化前后的性能变化，供实践报告使用。

### 6. 主要交付物

```text
src/backend/riscv_instruction.*
src/backend/code_generator.*
src/backend/frame_layout.*
src/backend/register_allocator.*
src/backend/peephole_optimizer.*
tests/backend/
tests/performance/
```

## 六、四个模块之间的接口约定

为了保证四人能够独立开发，需要在正式编码前确定以下接口。

### 1. A与B之间

A输出AST根节点：

```cpp
std::unique_ptr<CompUnitAST> parseProgram();
```

B只能读取和标注AST，不能依赖Parser内部状态。

### 2. B与C之间

B为每个变量、常量、函数和参数绑定唯一符号对象：

```cpp
struct IdentifierExpr {
    std::string name;
    Symbol* resolvedSymbol;
};
```

C通过`resolvedSymbol`访问对象，不再按照字符串名称搜索符号表。

### 3. C与D之间

C输出完整IR模块：

```cpp
struct IRModule {
    std::vector<IRGlobal> globals;
    std::vector<IRFunction> functions;
};
```

D只依赖IR公共接口，不直接访问AST和符号表实现。

### 4. 优化开关

由A解析命令行参数：

```text
compiler          不启用优化
compiler -opt     启用中端和后端优化
```

A将该配置传递给C和D，C执行中端优化，D执行寄存器分配和指令级优化。

## 八、开发阶段安排

### 第一阶段：接口和最小框架

四人共同确定：

1. AST节点结构。
2. 符号表中的Symbol结构。
3. IR指令集合。
4. RISC-V函数调用约定。
5. 项目目录和代码规范。

本阶段结束时，需要保证整个空框架能够正常编译。

### 第二阶段：各模块独立开发

A完成词法、语法和AST生成。

B可以先手工构造AST测试符号表和常量求值。

C可以先手工构造带符号信息的AST测试IR生成。

D可以先手工构造IR测试RISC-V汇编生成。

这样即使上游模块暂时未完成，下游成员也能够独立开发。

### 第三阶段：最小功能联调

建议按以下顺序逐步联调：

```text
return常量
→ 算术表达式
→ 局部变量
→ if-else
→ while
→ 函数调用
→ 递归
→ 全局变量和常量
→ 短路求值
```

每增加一种功能，都应保留对应的回归测试。

### 第四阶段：完整测试与优化

1. 补充复杂组合测试。
2. 对照gcc进行差分测试。
3. 开启`-opt`验证优化前后结果一致。
4. 优化寄存器分配和汇编指令。
5. 在全新环境中验证构建和运行命令。
6. 检查在线评测要求的标准输入输出接口。

## 九、实践报告分工

实践报告不建议完全交给一位成员编写，而应与代码模块对应：

| 成员 | 报告负责内容 |
|---|---|
| A | 总体架构、词法分析、语法分析、AST设计、工程构建 |
| B | 符号表、作用域、语义分析、常量处理、全局对象 |
| C | 中间代码设计、控制流构建、短路求值、中端优化 |
| D | RISC-V代码生成、栈帧、调用约定、寄存器分配、性能实验 |

每位成员还应提供：

1. 本模块设计目标。
2. 核心数据结构。
3. 关键算法或流程图。
4. 代表性代码说明。
5. 测试用例和测试结果。
6. 遇到的问题及解决方法。

最终由A统一工程名称、目录和图表编号，B检查语言和术语，C整理优化实验，D整理性能数据。这样报告工作同样能够平均分配。

## 十、代码评审与交叉检查

为了避免成员只熟悉自己的模块，可以采用交叉评审：

```text
A评审D的汇编输出和接口使用
B评审A的AST结构和声明节点
C评审B的语义标注是否满足IR需求
D评审C的IR是否便于后端翻译
```

每个合并请求至少由一位下游模块成员检查。例如A修改AST后，应由B和C确认不会影响语义分析和IR生成。

---

## 十一、工作量平衡说明

A不仅负责词法和语法分析，还承担AST设计、构建系统和自动化测试平台，因此不会出现前端工作量明显偏少的问题。

B除符号表和语义分析外，还负责常量表达式求值、全局变量和常量管理、函数信息及作用域管理，这些内容与本年度新增功能密切相关，工作量较为充足。

C需要完成IR设计、复杂控制流生成、短路逻辑以及多项中端优化，是整个编译器中逻辑复杂度较高的部分。

D负责RISC-V32代码生成、函数调用约定、栈帧、寄存器分配和性能优化。虽然难度较高，但D不需要处理AST、作用域和高级控制流语义，其输入已经被C转换为统一IR，因此总体工作量能够保持平衡。

通过以上划分，每位成员都负责一个完整、可展示、可独立测试的核心模块，同时模块之间通过固定接口连接，既能减少Git冲突，也有利于最终答辩时明确说明每位成员的实际贡献。
